## 1. Goal

Add a CI workflow that runs `bun install`, builds, type-checks, and tests on every PR and on
pushes to `main`, so each per-task PR is gated by the same green bar the PRDs require.

## 2. Why

The workflow is one PR per task with explicit Acceptance/Test plans. CI enforces those gates
automatically instead of by hand. Lands early (after scaffold) so tasks 03–08 are checked on
arrival. Optional only in that nothing depends on it — but it's the cheapest quality lever.

## 3. Scope

### In
- `.github/workflows/ci.yml`:
  - triggers: `pull_request`, `push` to `main`.
  - `oven-sh/setup-bun` (pin a bun version).
  - steps: `bun install --frozen-lockfile` → `bun --filter '*' build`
    (or per-package) → `bun --filter render-sdk test` → typecheck
    (`bun --filter render-sdk types` + `bun --filter docs types:check`).
  - cache bun install.
- A status summary so failing build/test/types blocks the PR.

### Out
- Publishing / release automation / changesets (future).
- Live tests against real AWS/Chromium (mocks only in CI).
- Matrix across OS/Node versions (single linux + pinned bun in v1).

## 4. Deliverables

```
.github/workflows/ci.yml
```

## 5. Workflow sketch

```yaml
name: ci
on:
  pull_request:
  push: { branches: [main] }
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
        with: { bun-version: <pin> }
      - run: bun install --frozen-lockfile
      - run: bun --filter render-sdk build
      - run: bun --filter render-sdk types
      - run: bun --filter render-sdk test
      - run: bun --filter docs build
```
> Exact `--filter` targets/script names confirmed against PRs 01/02. Package name for filter
> is `@remocn/render-sdk` (or alias) — verify the filter selector bun expects.

## 6. Acceptance criteria

- [ ] Workflow runs on a PR and on push to `main`.
- [ ] A failing build / test / typecheck **fails** the job (non-zero exit blocks merge).
- [ ] `--frozen-lockfile` catches lockfile drift.
- [ ] Remotion peers install in CI (devDeps) and adapter tests pass with mocks (no real
      Chromium/AWS).
- [ ] Run completes in a reasonable time with bun cache.

## 7. Test plan

- Self-validating: open the PR, confirm the Actions run is green; intentionally break a test
  locally to confirm CI would fail (don't merge the break).

## 8. Risks / notes

- Pin the bun version to avoid surprise breakage; bump deliberately.
- If `bun --filter` selector differs from the package `name`, adjust (bun filters by package
  name or path).
- Docs build needs `fumadocs-mdx` postinstall to have run — `bun install` handles it.

## 9. Execution (OMC)

`executor` writes the workflow; `verifier` confirms the gates fail on red. Git (and enabling
required checks on the repo) is yours.
