### Files created (packages/render-sdk/src/)

- `adapter.ts`: RenderAdapter<TOptions> SPI (start/getState/getUrl/download) + OptionsOf<A> helper.
- `index.ts`: kept VERSION="0.0.0"; re-exports the full public surface (type-only where types).
- Tests: handle.test.ts (round-trip both adapters, "." bucket, 10 garbage cases all -> not_found),

- `bun run build`: exit 0; dist now 11 files (index + 5 core .d.ts + server/lambda + 3 .js).

### Learnings / patterns for future PRDs
- Branded RenderHandle breaks `expect(handle).toBe("s~abc")` — bun's toBe infers `expected` from
  the branded actual (NoInfer<RenderHandle>), so a plain string literal isn't assignable. Cast the
  actual: `expect(handle as string).toBe(...)`. Runtime value is a normal string.
- tsc preserves JSDoc into .d.ts, so a comment containing the literal `@remotion` will trip a
  `grep "@remotion" dist`. Keep the "remotion-free" rationale in comments without the `@remotion`
  token (lowercase "remotion" is fine).
- decodeHandle uses split("~") with fixed segment counts (server=2, lambda=4) + a blank-field
  guard (trim().length===0) — this is what rejects `s~` and `l~r~ ~mp4` while letting `my.bucket`
  through. region/functionName intentionally NOT in the handle (come from adapter config).

  core types — do not redefine them.

## PRD 05 — RenderSdk + waitForCompletion — DONE (session a236fb66)

### What was implemented
The public `RenderSdk<A>` facade + the adapter-agnostic `waitForCompletion` poll loop. Both
remotion-free, verified against fake/scripted in-memory adapters (no real backend yet).

### Files created (packages/render-sdk/src/)
- `wait.ts`: standalone `waitForCompletion(adapter: Pick<RenderAdapter<unknown>,"getState">,
  handle, opts?)`. `for(;;)` poll loop: checks signal -> getState -> onProgress(progress) ->
  done returns final state / error throws RenderError("render_failed", state.error ?? "Render
  failed") / past deadline throws RenderError("timeout") -> abortableDelay(min(intervalMs,
  deadline-now), signal). `abortableDelay` is a setTimeout wrapped in a Promise with a once
  "abort" listener that clearTimeouts + rejects RenderError("adapter_error","aborted") — this is
  what makes abort interrupt MID-WAIT, not just between polls. DEFAULT_INTERVAL_MS = 1000.
- `sdk.ts`: `class RenderSdk<A extends RenderAdapter<any>>`, ctor({ adapter, store? }) stores both
  as readonly fields. start(input, options?: OptionsOf<A>) / getState / getUrl / download are
  one-line delegations to this.adapter; waitForCompletion(handle, opts?) delegates to the
  standalone loop with this.adapter. Exports `RenderSdkConfig<A>` type. The SDK never reads/writes
  the store — it only holds it for adapters (server adapter, PR 06, will use it).
- `index.ts`: added `export { RenderSdk, type RenderSdkConfig } from "./sdk"` and
  `export { waitForCompletion } from "./wait"`.

### Tests (test/)
- `wait.test.ts`: scripted getState-only adapter (queued->rendering(.5)->done) asserts onProgress
  [0,.5,1] + resolution; separate cases throw render_failed on error, timeout on timeoutMs, abort
  mid-wait via AbortController, and immediate reject on already-aborted AbortSignal.abort().
- `sdk.test.ts`: fakeAdapter records a discriminated `Call[]`; verifies all 5 methods delegate,
  store is held (sdk.store === store) but untouched (store.get null after start), store optional,
  and a `@ts-expect-error` case proving `start`'s 2nd arg narrows to OptionsOf<A> (wrong-shape
  `{ chromiumOptions: {} }` fails to compile).

### Verification evidence (all exit 0)
- `bun run types` (tsc --noEmit): 0 errors. `bun run build`: succeeded; dist now 15 files
  (+sdk.{js,d.ts}, +wait.{js,d.ts}). `bun test`: 39 pass, 0 fail (was 28). `grep -rl @remotion
  dist`: none. Architect reviewer: APPROVED against all PRD-05 acceptance criteria. Deslop pass:
  removed one redundant monotonic assertion in wait.test.ts; post-deslop regression still 39 pass.

### Learnings / patterns for future PRDs
- `@ts-expect-error` IS the type-narrowing test: because tsc errors on an UNUSED @ts-expect-error
  directive, a green `bun run types` proves the wrong-shape option genuinely fails to compile. No
  separate expectTypeOf/.test-d harness needed — the directive self-verifies under the existing
  types gate (test/ is in tsconfig include).
- waitForCompletion is intentionally `Pick<RenderAdapter,"getState">`, not the full adapter — keeps
  it trivially testable with a one-method fake and documents that polling only needs getState.
- Error contract is THROW (not resolve): done -> resolve(state); error -> throw render_failed;
  timeout -> throw timeout; abort -> throw adapter_error/"aborted". getState itself always resolves
  a state object; the loop is what converts terminal states into resolve/throw.
- RenderSdk is a pure thin delegator: it holds `store` but never calls it. Store writes are owned by
  the server adapter (PR 06), which receives the store via its OWN config, not via the SDK. Confirm
  this wiring direction when implementing PR 06 — do not make the SDK seed the store.
- Next: PR 06 (server adapter, depends 03/04/05) + PR 07 (lambda adapter, depends 03/05) are now
  unblocked and independent — parallel branches. Both plug into a real `new RenderSdk({ adapter })`.

## PRD 06 — Server Adapter (`@remocn/render-sdk/server`) — DONE (session 90eeeca7)

### What was implemented
`RenderServer(config)` — the in-process adapter over `@remotion/renderer` with a `p-limit`
limiter, store-backed progress, disk output, and getUrl/download from disk. Remotion stays an
optional peer dep imported only by the /server subpath (root export rema­ins remotion-free).

### Files created / changed (packages/render-sdk/)
- `src/server/index.ts`: RenderServer(config: ServerConfig): RenderAdapter<ServerOptions>.
  ServerConfig{serveUrl, workDir(req), publicUrl?, concurrency?(def 2), store?(def fresh
  InMemoryStore)}; ServerOptions{concurrency?, chromiumOptions?, timeoutInMilliseconds?}.
  start() → codec(default h264) → encodeServerHandle(crypto.randomUUID()) → store.create(queued)
  → `void limit(() => runRender(...))` DETACHED → returns handle. runRender: store.update(rendering)
  → selectComposition → mkdir -p workDir → renderMedia({outputLocation:`${workDir}/${handle}.${ext}`,
  onProgress→store.update(clamp01(p.progress))}) → store.update(done,1); catch → store.update(error,
  message, errorCode=classify). getState maps record→{status,progress,error?}, missing→not_found.
  getUrl: publicUrl ? `${publicUrl}/${handle}.${ext}` : `${handle}.${ext}` (trailing slash trimmed),
  ext from stored codec. download: stat file → Readable.toWeb(createReadStream) as unknown as
  ReadableStream; missing record/file → not_found.
- `src/store/types.ts`: additive optional `errorCode?: RenderError["code"]` on RenderRecord (the
  channel that surfaces version_mismatch vs render_failed — RenderState has no code field).
- Tests (6 files + shared `test/_server-harness.ts`): server.start / .limiter / .url / .download /
  .error / .types. Harness mocks `@remotion/renderer` via `mock.module` + dynamic
  `await import("../src/server")`, with mutable `remotion.selectImpl/renderImpl` swapped per test,
  plus deferred()/flush()/makeWorkDir()/seedDone()/expectRenderErrorCode() helpers.

### Learnings / patterns for future PRDs
- Mocking a statically-imported native peer dep in bun: register `mock.module("@remotion/renderer",
  factory)` at the TOP of the test file, THEN `const { RenderServer } = await import("../src/server")`
  — static top imports run before top-level code, so the adapter must be dynamically imported AFTER
  the mock. Factory delegates to a mutable `remotion` object so each test swaps impls without
  re-registering. Each test file registers its own mock.module; works across the 13-file suite.
- `Readable.toWeb()` returns a `node:stream/web` ReadableStream that TS treats as nominally distinct
  from the global `ReadableStream` (getReader overload arity differs) — `as unknown as ReadableStream`
  is the sanctioned bridge; identical at runtime.
- remotion v4 (4.0.484): ENABLE_V5_BREAKING_CHANGES=false so licenseKey is OPTIONAL — renderMedia
  needs only {codec, composition, serveUrl}. selectComposition returns Promise<VideoConfig> directly.
  remotion's validPixelFormats == our PixelFormat union exactly, and our Codec ⊂ remotion Codec, so
  input.codec/pixelFormat forward without casts (codec type-guard test locks this).
- Detached render: failures are caught INSIDE runRender and written to the store, so the unawaited
  `limit(...)` promise never rejects (no unhandledRejection). Concurrency is provable deterministically
  by gating renderMedia with a deferred() under concurrency:1 — 2nd job's runRender never starts, so
  its record stays `queued`.
- errorCode lives on RenderRecord, not RenderState (kept the public poll shape unchanged). version
  classification is reactive: /version/i && /match/i on the message → version_mismatch, else
  render_failed (matches the locked design record — no proactive version probing).

  adapters; PR 09 (CI) can land anytime after 02.
